# Design System To MP Rich HTML Prompt

> 本提示词用于把“文章 slug + 文章正文”转换成一篇图文并茂、可复制到微信公众号编辑器的 Hugo `mp.html`。视觉规范来自文章目录下的 `design-system.png`，公众号兼容规则以指定 skill 为准，不在本文档重复维护。

## 输入

只需要两个输入：

1. `article_slug`：文章目录名，英文 kebab-case，例如 `city-guide`。
2. `article_body`：完整文章正文，可以是 Markdown、纯文本、列表或表格。

## 派生路径

根据 `article_slug` 自动派生：

- 视觉规范图：`../content/{article_slug}/design-system.png`
- 输出文件：`../content/{article_slug}/mp.html`
- 图片目录：`../content/{article_slug}/assets/`
- Hugo 发布后的图片引用：`/{article_slug}/assets/{filename}`
- 文章标题：从 `article_body` 的一级标题或内容主题中提炼

## 必须使用的 Skill

- 使用 [$hxgdzyuyi-skills:mp-rich-html](../../../mine-work/skills/hxgdzyuyi-skills/skills/mp-rich-html/SKILL.md) 作为微信公众号 HTML 兼容规则的唯一权威来源。
- 如需生成、修复或统一配图，使用 [$imagegen](../../../.codex/skills/.system/imagegen/SKILL.md)，并把最终图片保存到派生图片目录。

不要在本提示词中重新展开公众号 HTML 过滤规则；执行时直接遵守 `mp-rich-html` skill 的最新要求。

## 核心任务

根据 `article_body` 推导文章结构、章节节奏、配图计划和图文卡片；根据 `design-system.png` 提取视觉语言，并将两者结合成微信公众号兼容的富文本 HTML，写入派生输出文件。

重要边界：

- `design-system.png` 决定视觉语言，不决定文章主题。
- `article_body` 决定文章内容、章节和图片主题。
- 不要照搬设计系统图里的示例城市、示例标题、示例卡片内容或示例图片主题，除非它们和 `article_body` 明确一致。
- 不能只做文字卡片；图文密度应参考设计系统图中的 demo 节奏。

## 执行流程

1. 分析 `article_body`
   - 提炼文章标题、导语、章节、小标题、列表、表格、重点句、链接和结尾。
   - 识别适合视觉化的实体：人物、地点、食物、物品、产品、设备、植物、事件、流程、地图、场景、数据图解等。
   - 判断哪些内容适合纯文本、哪些适合图文卡片、哪些适合提示框或表格卡片。

2. 分析 `design-system.png`
   - 提取配色、字体层级、标题样式、卡片样式、分隔元素、图片容器和整体版式节奏。
   - 提取图像风格，例如水彩、纸张颗粒、复古、摄影、拼贴、科技、极简、手绘、城市山水等。
   - 识别 demo 的图文密度：主视觉、章节图、内容卡缩略图、局部插画、装饰图标如何交替出现。

3. 制定配图计划
   - 先根据正文自动生成图片清单，再制作或引用图片。
   - 图片主题必须从正文推导，不要预设某篇文章的固定图片清单。
   - 每张图片都要服务相邻内容：主视觉、章节图、卡片缩略图、流程图或局部氛围图。

4. 生成素材并写入 HTML
   - 图片保存到派生图片目录。
   - HTML 图片引用使用 Hugo 发布路径：`/{article_slug}/assets/{filename}`。
   - 根据 `mp-rich-html` skill 生成可复制到公众号编辑器的内联 HTML。

5. 浏览器验证
   - 运行 Hugo 预览，打开 `/mp/mp-{article_slug}.html`。
   - 检查页面渲染、图片加载、移动端宽度、图文密度和公众号正文可复制区域。

## 配图推导规则

图片数量按文章长度和结构推导：

- 短文：少于 800 个中文字符，或少于 3 个主要章节，至少 3 张，目标 4-5 张。
- 中等文章：800-2000 个中文字符，或 3-5 个主要章节，至少 5 张，目标 6-8 张。
- 长文：超过 2000 个中文字符，或超过 5 个主要章节 / 大量列表，至少 8 张，目标 10-14 张。
- 如果 `design-system.png` 的 demo 是高图文密度风格，优先靠近目标数量，而不是只满足最低数量。

图片类型按内容推导：

- `hero-{article_slug}.png`：顶部主视觉，概括文章主题和情绪。
- `section-{section_slug}.png`：章节图，用于内容差异明显的主要章节。
- `card-{item_slug}.png`：内容卡缩略图，用于具体实体或列表条目。
- `diagram-{concept_slug}.png`：流程、结构、地图、对比或数据图解。
- `detail-{subject_slug}.png`：局部氛围图或物件图，用于增强节奏但必须贴合正文。

图片分布规则：

- 顶部必须有 1 张主视觉。
- 每个信息量较大的主要章节，优先安排 1 张章节图或 2-3 张卡片缩略图。
- 至少三分之一的图片应作为具体内容卡片缩略图，而不是全部作为全宽横幅。
- 不要把所有图片都做成同一种版式。
- 不要为了凑数量生成和正文无关的装饰图。
- 如果文章内容抽象，图片可以转为隐喻性场景、流程图、材料/纹理细节或概念图解，但必须能从正文解释其用途。

## 图片制作规则

- 如果 `design-system.png` 中有可复用的无文字局部元素，例如纹理、装饰、图标、无字插画或图片风格样本，可裁剪保存到图片目录。
- 如果 `design-system.png` 中的图像带有主要文字、示例文章内容、示例标签或与当前文章主题不匹配，不要直接裁剪使用；只把它作为风格参考。
- 当前文章需要的主视觉、章节图和内容卡缩略图，优先用 `imagegen` 按设计系统风格生成无文字图片。
- 生成图片必须继承 `design-system.png` 的视觉语言：配色、纹理、边框、光影、插画/摄影气质、构图密度和留白节奏。
- 禁止生成带有标题、正文、标签、Logo、水印或乱码文字的图片，除非用户明确要求某张图含有文字。

## 文章结构生成规则

根据 `article_body` 自动生成结构，不要硬套固定目录。

建议结构：

1. 顶部标题区：标题、短导语、关键词或摘要、主视觉。
2. 快速导览：从正文自动提炼 3-6 个阅读关键词或重点。
3. 主体章节：按正文自然结构重排，每章可包含段落、列表、图文卡片、引用、提示框或表格卡片。
4. 复杂列表：优先改成移动端稳定的卡片流；列表条目有具体实体时，可配卡片缩略图。
5. 表格：若列多或内容长，优先转为逐项卡片；若保留表格，按 `mp-rich-html` skill 的表格规则处理。
6. 结尾：总结、行动提示、路线建议、注意事项或延伸阅读，按文章内容决定。

不得删除用户正文中的核心信息点。可以为了公众号阅读体验改写小标题、合并相近段落、拆分长列表，但不能新增无法从正文推导的事实。

## 写入要求

输出文件写入：

```text
../content/{article_slug}/mp.html
```

素材写入：

```text
../content/{article_slug}/assets/
```

front matter 至少包含：

```yaml
title: "{article_title}"
type: "mp"
layout: "single"
url: "/mp/mp-{article_slug}.html"
```

如果生成了主视觉，添加：

```yaml
cover: "/{article_slug}/assets/hero-{article_slug}.png"
```

正文 HTML 的具体写法、元素选择、内联样式、图片样式和多列布局规则，以 `mp-rich-html` skill 为准。

## 生成前计划

写文件前先形成内部执行计划：

1. 设计系统摘要：颜色、字体层级、卡片、图片风格、图文密度。
2. 文章结构摘要：标题、章节、重点信息、可视化机会。
3. 配图计划：图片数量、每张图片的用途、文件名、生成/裁剪方式、放置位置。
4. 公众号组件计划：主视觉、导览卡、章节标题、图文卡片、提示框、列表或表格替代方案。

这份计划用于指导实现，不需要原样写进 `mp.html`。

## 验收重点

- Hugo 预览页可正常打开。
- `#js_content` 内是公众号正文主内容。
- 页面遵守 `mp-rich-html` skill 的最新自查清单。
- 图片数量符合“配图推导规则”，且全部可加载。
- 图片 `src` 使用 `/{article_slug}/assets/...`、真实 `https://` 或 `data:image/...`。
- 至少三分之一图片是具体内容卡片缩略图，而不是全部作为章节横幅图。
- 单栏和双栏模块在 375px、414px、578px 宽度下不溢出、不遮挡。
- 正文不是由整张长图组成，主要文字均为真实 HTML 文本。

## 最终说明

最终说明必须包含：

- 输出 HTML 文件路径和预览 URL。
- 采用的设计系统摘要。
- 文章结构如何由 `article_body` 推导。
- 图片清单：文件名、用途、生成/裁剪方式、引用路径。
- 如果裁剪图片，记录来源区域：`x`、`y`、`width`、`height`。
- 如果 imagegen 生成图片，记录提示词摘要。
- 浏览器渲染验证结果。
- 公众号发布时图片是否需要经复制脚本内联，或替换为素材库 HTTPS 地址。

## 关键原则

- 只有 `article_slug` 和 `article_body` 是输入；其他内容都从 slug、正文和项目约定中派生。
- `design-system.png` 决定视觉语言，不决定文章主题。
- 图文密度应模仿设计系统 demo 的节奏，但图片主题必须来自当前文章。
- 图文并茂不等于堆图；每张图都要服务相邻内容。
- 公众号兼容性由 `mp-rich-html` skill 负责，本提示词只负责通用编排和图文策略。
